* Overview

SteamPulse is an AI-powered Steam game intelligence platform at *steampulse.io*.

- *Public site*: AI-synthesized review reports for all Steam games with any reviews. SEO-driven,
  ad-free, cross-linked. Hundreds of genre/tag/developer index pages for long-tail SEO.
- *Premium layer*: Developer-focused. Custom analysis, fresher data, action items, refund risk
  signals, NL chat over full catalog.

One Python codebase. AWS for everything. Feature flags separate tiers.

* Tech Stack

|----------+----------------------------------------------------------------------------------------|

* Hosting Architecture

** Frontend / Backend split

Two Lambda functions behind one CloudFront distribution:

- *Next.js app* (OpenNext → Lambda): all UI, SSR/ISR, React components. Calls FastAPI internally.
- *FastAPI app* (Lambda Web Adapter → Lambda): all business logic — LLM analysis, Steam data
  fetching, payment validation, V2 chat. JSON API only, no HTML.

#+BEGIN_SRC
CloudFront (steampulse.io)
  ├── /static/*   → S3 (Next.js + FastAPI static assets — versioned filenames)
  ├── /api/*      → FastAPI Lambda Function URL
  └── /*          → Next.js Lambda (via OpenNext)
#+END_SRC

OpenNext: https://opennext.js.org/ — canonical way to self-host Next.js on AWS without Vercel.

** Page rendering strategy

| Page type      | Next.js strategy       | Notes                                          |
|----------------+------------------------+------------------------------------------------|
| Game report    | SSR (on-demand)        | CloudFront caches 24h + stale-while-revalidate |
| Genre index    | ISR (revalidate daily) | Aggregated insights, updated nightly           |
| Tag index      | ISR (revalidate daily) |                                                |
| Developer page | ISR (revalidate daily) |                                                |
| Home           | ISR                    | Trending/featured games                        |
| Compare        | SSR (on-demand)        | Dynamic game pair                              |

** Template & content updates

| Update type             | Mechanism                                             | Time to global |
|-------------------------+-------------------------------------------------------+----------------|
| Code/template change    | Deploy new Lambda + =/*= wildcard invalidation (free) | ~1 min         |
| Single game re-analysis | Targeted =/games/{appid}/*= invalidation              | ~30 sec        |
| Featured spot purchase  | Write to CloudFront KeyValueStore                     | Seconds        |

** Featured spots via CloudFront KeyValueStore

KVS holds sponsor data as JSON. CloudFront Function reads it at edge and injects a sponsor
banner into the HTML response. No Lambda invocation, no cache invalidation needed.

** Lambda Web Adapter (FastAPI)

Official AWS Labs project: https://github.com/awslabs/aws-lambda-web-adapter
FastAPI runs natively — zero code changes. One line in Dockerfile:

#+BEGIN_SRC dockerfile
FROM public.ecr.aws/awsguru/aws-lambda-adapter:0.8.4 AS lambda-adapter
FROM public.ecr.aws/lambda/python:3.12
COPY --from=lambda-adapter /lambda-adapter /opt/extensions/lambda-adapter
ENV PORT=8080
#+END_SRC

Use Lambda Function URLs (not API Gateway) to avoid the 29-second timeout limit.
Initialize DB connections OUTSIDE the handler for connection reuse on warm invocations.

* GitHub → AWS Pipeline (CDK Pipelines)

** How it works

AWS CDK Pipelines listens to your GitHub repo via a *CodeStar Connection* (a GitHub App
installed into your AWS account). On every push to =main=, GitHub notifies AWS CodePipeline
via webhook and the pipeline runs automatically.

#+BEGIN_SRC
GitHub (your account)
  └── push to main
       └── CodeStar Connection (GitHub App) notifies AWS
            └── CodePipeline kicks off
                 ├── Source: pulls code from GitHub
                 ├── Build: cdk synth
                 ├── Deploy → staging (auto)
                 └── Deploy → production (manual approval gate)
#+END_SRC

** One-time setup (manual, done once)

1. In AWS Console → *Developer Tools → Connections* → Create connection → GitHub
2. Click "Install GitHub App" → authorize access to your repo
3. Copy the Connection ARN → store in CDK context or SSM Parameter

That's it. After that, every =git push= triggers the pipeline automatically.

** CDK Pipelines source (in code)

#+BEGIN_SRC python
source = CodePipelineSource.connection(
    "your-github-username/steam-pulse",
    "main",
    connection_arn="arn:aws:codestar-connections:us-east-1:ACCOUNT:connection/UUID",
)
#+END_SRC

The connection ARN is the only thing that ties CDK to your specific GitHub account.
Store it in =cdk.json= context or AWS SSM — never hardcode in source.

** Self-mutating pipeline

After the *first* manual =cdk deploy= (to bootstrap the pipeline itself), all future
infrastructure changes are deployed automatically when you push to =main=. You never
run =cdk deploy= again unless bootstrapping a new environment.

** CDK best practices (mandatory)

- No physical resource names (let CDK generate to avoid naming conflicts)
- No env var lookups inside constructs — pass as props or CDK context
- Secrets in AWS Secrets Manager, referenced by ARN
- Stateful stacks (=data_stack=) have =termination_protection=True=
- =CodePipelineSource.connection()= for GitHub (CodeStar Connections, not PAT)
- Bootstrap with =--termination-protection= flag
- Run =cdk deploy= manually once; pipeline self-mutates thereafter

* Crawler Architecture

** Data sources

SteamPulse needs three upstream sources, accessed via a =SteamDataSource= abstraction
(same pattern as =BaseStorage= — swap implementations without touching crawler logic).

| Source           | URL                                        | Data                          |
|------------------+--------------------------------------------+-------------------------------|
| Steam App List   | api.steampowered.com/ISteamApps/GetAppList | All ~100k app IDs + names     |
| Steam Store API  | store.steampowered.com/api/appdetails      | Game metadata, genres, price  |
| Steam Reviews    | store.steampowered.com/appreviews/{appid}  | Review text, thumbs, playtime |
| SteamSpy         | steamspy.com/api.php                       | Tags, owner count estimates   |

** SteamDataSource abstraction

The abstraction exists so the data source can be swapped without touching crawler logic
(e.g. direct Steam API today, a different provider later if rate limits become a problem).

#+BEGIN_SRC python
class SteamDataSource(ABC):
    async def get_app_list(self) -> list[dict]: ...
    async def get_app_details(self, appid: int) -> dict: ...
    async def get_reviews(self, appid: int, max_reviews: int) -> list[dict]: ...
    async def get_steamspy_data(self, appid: int) -> dict: ...

class DirectSteamSource(SteamDataSource):
    """Default. Calls Steam Store API and SteamSpy directly."""
#+END_SRC

** Two crawler Lambda functions

1. *app-crawler*: Crawls game metadata from Steam Store API + SteamSpy. ~100k games.
   - SQS queue: =app-crawl-queue= (batch size 10)
   - Rate: ~200 req/5min per IP — use jitter to avoid bans

2. *review-crawler*: Fetches reviews for all games with at least 1 review.
   - SQS queue: =review-crawl-queue= (batch size 1 — each game is heavyweight)
   - Rate: paced by SQS visibility timeout (~2 min max per game)

** Step Functions: Analysis Pipeline

State machine triggered after review crawl completes for each game:

#+BEGIN_SRC
StartAnalysis
  └── PrepareChunks   (Lambda: split reviews into 50-review batches)
       └── Map (MaxConcurrency: 5)
            └── AnalyzeChunk  (Lambda: Haiku — extract themes per chunk)
       └── SynthesizeReport  (Lambda: Sonnet — merge chunks → structured JSON)
            └── StoreReport   (Lambda: upsert to RDS reports table)
                 └── InvalidateCache  (Lambda: targeted CloudFront invalidation)
#+END_SRC

Step Functions Express Workflows. Cost: ~$0.30 for entire 6k seed crawl (negligible).
Map(MaxConcurrency=5) means a 2,000-review game (40 chunks) runs in ~8 parallel batches.

* Bootstrap / Generation Strategy

Games are analyzed in descending review count order. Three phases:

1. *Phase 1 — Top 500* (highest review count): pre-generate immediately after first crawl.
   Warms CloudFront cache. Covers ~80% of search traffic. ~$170 upfront.
2. *Phase 2 — Remaining 500+ review games* (~5,500 games): batch-process in background
   after Phase 1, or lazy on first request.
3. *Phase 3 — Long tail* (1–499 reviews, ~90k+ games): lazy on first request only.

All games get a report page. Low-review games get a "Limited data" badge and shorter analysis
(fewer chunks = less LLM cost). Cost scales with review count, not a flat fee (~$0.34/game max).

Lazy generation flow:
- First request for unanalyzed game → FastAPI triggers Step Functions → returns status page
- Client polls =/api/status/{job_id}= every 5s → delivers full report when analysis completes

* Data Model (PostgreSQL)

-- All Steam games (~100k apps from crawl)
CREATE TABLE games (

  playtime_hours  INTEGER,

  window_start TIMESTAMPTZ DEFAULT NOW()

* Site Architecture

** URL structure

/compare/{appid1}/{appid2}          # Head-to-head comparison

** SEO requirements

- H1 = game name always
- H2s = "What Players Love", "Common Complaints", "Who Should Play This"
- Next.js SSR guarantees Googlebot sees full content on first request
- Structured data: =VideoGame= schema + =Review= aggregate
- Canonical URLs, 301 redirects on slug changes
- Sitemap with =<lastmod>= set to =last_analyzed= date
- OG tags: game cover image + one-liner

** Cross-linking strategy

Three directions from every page:
1. *Horizontal*: competitive_mentions → bidirectional game→game links
2. *Up*: breadcrumb to genre/tag index
3. *Down*: genre/tag/developer pages link back to all games in that group

** Content moat: index pages

Genre, tag, and developer pages aggregate insights across all games in that group:
- "The #1 complaint in Survival Horror is checkpoint frequency (37% of games)"
- Hundreds of additional high-value pages beyond the game report pages.

* Monetization

** Decision: No display ads

- Gaming audience has 40-60% ad-block rate
- AdSense CPMs for gaming: $1-3 (not meaningful until 500k+ monthly visits)
- Ads undermine premium conversion at exactly the wrong moment
- The public site is a *sales funnel*, not a profit center

** Revenue streams (in priority order)

| Stream                    | Audience           | Type          | Notes                                      |
|---------------------------+--------------------+---------------+--------------------------------------------|
| Steam affiliate links     | Gamers             | Passive       | ~5% on referred purchases, zero friction   |
| Per-analysis credits      | Developers         | Transactional | $7 single / $15 for 5-pack (Lemon Squeezy) |
| Pro subscription ($49/mo) | Indie studios      | Recurring     | V2 — NL chat over full catalog             |
| Sponsored genre/tag pages | Devs wanting reach | Direct sales  | $10-50 CPM equivalent, post-launch         |

** Premium CTA placement

1. After the Complaints section: "8 developer action items ranked by frequency →"
2. Genre index pages: "Analyze your own game in this genre" at top
3. Bottom of free report: "You're a developer? Run this on your own game."

CTAs always speak to *developers*, never to gamers. Gamers won't pay. Developers will.

- ❌ "Unlock the full report — $7"
- ✅ "Get action items, refund signals, and feature gaps your competitors haven't fixed — $7"
