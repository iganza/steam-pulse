* Overview

SteamPulse is an AI-powered Steam game intelligence platform at *steampulse.io*.

- *Public site*: AI-synthesized review reports for all Steam games with any reviews. SEO-driven,
  ad-free, cross-linked. Hundreds of genre/tag/developer index pages for long-tail SEO.
- *Premium layer*: Developer-focused. Custom analysis, fresher data, action items, refund risk
  signals, NL chat over full catalog.

One Python codebase. AWS for everything. Feature flags separate tiers.

* Hosting Architecture

** Frontend / Backend split

Two Lambda functions behind one CloudFront distribution:

- *Next.js app* (OpenNext → Lambda): all UI, SSR/ISR, React components. Calls FastAPI internally.
- *FastAPI app* (Lambda Web Adapter → Lambda): all business logic — LLM analysis, Steam data
  fetching, payment validation, V2 chat. JSON API only, no HTML.

#+BEGIN_SRC
CloudFront (steampulse.io)
  ├── /static/*   → S3 (Next.js + FastAPI static assets — versioned filenames)
  ├── /api/*      → FastAPI Lambda Function URL
  └── /*          → Next.js Lambda (via OpenNext)
#+END_SRC

OpenNext: https://opennext.js.org/ — canonical way to self-host Next.js on AWS without Vercel.

** Page rendering strategy

| Page type      | Next.js strategy         | Notes                                          |
|----------------+--------------------------+------------------------------------------------|
| Game report    | SSR (on-demand)          | CloudFront caches 24h + stale-while-revalidate |
| Genre index    | ISR (revalidate daily)   | Aggregated insights, updated nightly           |
| Tag index      | ISR (revalidate daily)   |                                                |
| Developer page | ISR (revalidate daily)   |                                                |
| Home           | =force-dynamic= (no ISR) | See note below — must not pre-render           |
| Trending       | =force-dynamic= (no ISR) | See note below — must not pre-render           |
| Compare        | SSR (on-demand)          | Dynamic game pair                              |

*** Why homepage and trending use force-dynamic (not ISR)

=API_URL= is a Lambda runtime env var set by CDK at deploy time. It is *not* available in the
CodeBuild environment that runs =next build=. ISR pages are pre-rendered at build time — if
=API_URL= is unset, =getGames()= fails silently (caught by =Promise.allSettled=), the page
renders with no games, and that empty render gets baked into the Lambda bundle.

=force-dynamic= skips build-time pre-rendering entirely. Every request renders in the Lambda
where =API_URL= is set. Do not change these pages back to ISR without also injecting =API_URL=
into the CodeBuild environment (circular dependency — the URL is only known after the API Lambda
is deployed).

** Template & content updates

| Update type             | Mechanism                                             | Time to global |
|-------------------------+-------------------------------------------------------+----------------|
| Code/template change    | Deploy new Lambda + =/*= wildcard invalidation (free) | ~1 min         |
| Single game re-analysis | Targeted =/games/{appid}/*= invalidation              | ~30 sec        |
| Featured spot purchase  | Write to CloudFront KeyValueStore                     | Seconds        |

** Featured spots via CloudFront KeyValueStore

KVS holds sponsor data as JSON. CloudFront Function reads it at edge and injects a sponsor
banner into the HTML response. No Lambda invocation, no cache invalidation needed.

** Image handling (next/image)

=next/image= with external URLs normally proxies through =/_next/image= for optimization.
OpenNext deploys this as a *separate* Lambda (=image-optimization-function/=) — it is NOT
part of the main SSR Lambda. As of current deployment, this function is not deployed.

*Current approach*: =images: { unoptimized: true }= in =next.config.ts=. This renders a plain
=<img>= tag pointing directly at the Steam CDN URL — no proxy, no optimization Lambda needed.
Steam CDN images are already appropriately sized (header images are 460×215px).

*If optimization is ever needed* (WebP conversion, responsive resizing): deploy
=frontend/.open-next/image-optimization-function/= as a separate Lambda and add a
=/_next/image*= CloudFront behaviour routing to it with query string forwarding for
=url=, =w=, =q= params.

** Lambda Web Adapter (FastAPI)

Official AWS Labs project: https://github.com/awslabs/aws-lambda-web-adapter
FastAPI runs natively — zero code changes. One line in Dockerfile:

#+BEGIN_SRC dockerfile
FROM public.ecr.aws/awsguru/aws-lambda-adapter:0.8.4 AS lambda-adapter
FROM public.ecr.aws/lambda/python:3.12

Use Lambda Function URLs (not API Gateway) to avoid the 29-second timeout limit.
Initialize DB connections OUTSIDE the handler for connection reuse on warm invocations.

* GitHub → AWS Pipeline (CDK Pipelines)

** How it works

AWS CDK Pipelines listens to your GitHub repo via a *CodeStar Connection* (a GitHub App
installed into your AWS account). On every push to =main=, GitHub notifies AWS CodePipeline
via webhook and the pipeline runs automatically.

#+BEGIN_SRC
GitHub (your account)
  └── push to main
       └── CodeStar Connection (GitHub App) notifies AWS
            └── CodePipeline kicks off
                 ├── Source: pulls code from GitHub
                 ├── Build: cdk synth
                 ├── Deploy → staging (auto)
                 └── Deploy → production (manual approval gate)
#+END_SRC

** One-time setup (manual, done once)

1. In AWS Console → *Developer Tools → Connections* → Create connection → GitHub
2. Click "Install GitHub App" → authorize access to your repo
3. Copy the Connection ARN → store in CDK context or SSM Parameter

That's it. After that, every =git push= triggers the pipeline automatically.

** CDK Pipelines source (in code)

#+BEGIN_SRC python
source = CodePipelineSource.connection(
    "your-github-username/steam-pulse",
    "main",
    connection_arn="arn:aws:codestar-connections:us-east-1:ACCOUNT:connection/UUID",
)
#+END_SRC

The connection ARN is the only thing that ties CDK to your specific GitHub account.
Store it in =cdk.json= context or AWS SSM — never hardcode in source.

** Self-mutating pipeline

After the *first* manual =cdk deploy= (to bootstrap the pipeline itself), all future
infrastructure changes are deployed automatically when you push to =main=. You never
run =cdk deploy= again unless bootstrapping a new environment.

* Crawler Architecture

** Data sources

SteamPulse uses Steam's own APIs directly via a =SteamDataSource= abstraction
(same pattern as =BaseStorage= — swap implementations without touching crawler logic).

| Source           | URL                                        | Data                          |
|------------------+--------------------------------------------+-------------------------------|
| Steam App List   | api.steampowered.com/ISteamApps/GetAppList | All ~100k app IDs + names     |
| Steam Store API  | store.steampowered.com/api/appdetails      | Game metadata, genres, categories, price, review counts |
| Steam Reviews    | store.steampowered.com/appreviews/{appid}  | Review text, thumbs, playtime |

Steam's own APIs provide all required fields (genres, categories, review counts, metadata).
No third-party data sources are used.

** SteamDataSource abstraction
#+BEGIN_SRC python
class SteamDataSource(ABC):
    async def get_app_list(self) -> list[dict]: ...
    async def get_app_details(self, appid: int) -> dict: ...
    async def get_reviews(self, appid: int, max_reviews: int) -> list[dict]: ...

class DirectSteamSource(SteamDataSource):
    """Default. Calls Steam Store API directly."""
#+END_SRC

** Two crawler Lambda functions

1. *app-crawler*: Crawls game metadata from Steam Store API. ~100k games.
   - SQS queue: =app-crawl-queue= (batch size 10)
   - Rate: ~200 req/5min per IP — use jitter to avoid bans

2. *review-crawler*: Fetches reviews for all games with at least 1 review.
   - SQS queue: =review-crawl-queue= (batch size 1 — each game is heavyweight)
   - Rate: paced by SQS visibility timeout (~2 min max per game)

** LLM execution paths

Two paths exist for the two-pass pipeline, sharing the same prompt logic:

| Path       | When used                    | Mechanism                          | Cost saving |
|------------+------------------------------+------------------------------------+-------------|
| Real-time  | On-demand single game        | Bedrock Converse API (boto3)       | None        |
| Batch      | Bulk seed, scheduled refresh | Bedrock Batch Inference + S3 JSONL | ~50%        |

*Real-time* uses =bedrock_runtime.converse()= — model-agnostic, swap model ID via env var, zero
code changes. Prompt caching (ephemeral cache points) works on this path.

*Batch* submits a JSONL file to S3, creates a Bedrock Batch job, and a Step Functions STANDARD
workflow polls every 5 minutes until completion. Two sequential jobs: Pass 1 (Haiku, all chunks
across all games in one job) → Pass 2 (Sonnet, one record per game). Prompt caching is NOT
supported in Batch Inference — remove =cache_control= from batch JSONL records.

Full design spec: =scripts/prompts/bedrock-batch-analysis.md=

** Step Functions Analysis Pipeline

#+BEGIN_SRC
StartAnalysis
  └── PrepareChunks   (Lambda: split reviews into 50-review batches + fetch game description)
       └── Map (MaxConcurrency: 5)
            └── AnalyzeChunk  (Lambda: Haiku via Converse API — extract themes per chunk)
       └── SynthesizeReport  (Lambda: Sonnet via Converse API — merge chunks → structured JSON)
            └── StoreReport   (Lambda: upsert to RDS via ReportRepository)
                 └── InvalidateCache  (Lambda: targeted CloudFront invalidation)
#+END_SRC

Express Workflows for real-time (single game). STANDARD Workflows for batch (long-running, hours).



*** Game description as silent synthesis context (POC)

The game's Steam description (=short_description= + =about_the_game=, HTML-stripped) is passed
to the *Pass 2 Sonnet synthesis prompt only* — not to the Pass 1 Haiku chunk calls.

Rationale:
- Already crawled — free, already in DB
- Zero extra cost — additional context in the same synthesis call, not a new LLM call
- Improves =one_liner=, =audience_profile=, =genre_context= quality without any new UI sections
- Enables implicit expectation mismatch detection (e.g., "claims deep story, players say thin
  narrative") — surfaces naturally in existing free sections
- Explicit description intelligence UI (Store Page Intelligence, Tag Optimization etc.) deferred to V2 Pro

Implementation: strip HTML with =re.sub(r'<[^>]+>', ' ', text)= before passing to LLM.
Cap at ~1,000 tokens to avoid inflating Pass 2 cost meaningfully (~$0.001/game extra at most).

* Code Architecture

** Repository → Service → Handler (mandatory pattern)

Every data access follows a strict three-layer pattern. *Nothing outside a Repository ever writes SQL.*

#+BEGIN_SRC
Handler (Lambda / FastAPI route)
  └── calls Service methods   (business logic only)
        └── calls Repository  (SQL I/O only)
#+END_SRC

- *Repository* (=library_layer/repositories/=): pure SQL I/O. One class per domain entity
  (=GameRepository=, =ReviewRepository=, =ReportRepository=, =CatalogRepository=, =JobRepository=).
  No business logic, no HTTP, no LLM. Methods return domain models or raise.
- *Service* (=library_layer/services/=): business logic. Coordinates repositories, calls Steam API,
  calls Bedrock. No raw SQL — call a repository if you need data.
- *Handler* (=lambda_functions/*/handler.py=): thin dispatcher. Parse input → call service → return.
  No SQL, no business logic.

** DRY principle (utils/)

Any logic used by more than one repository or service lives in =library_layer/utils/=:

| Utility         | Location           | Used by                            |
|-----------------+--------------------+------------------------------------|
| =slugify()=     | =utils/text.py=    | GameRepository, index page builder |
| =send_sqs_batch()= | =utils/aws.py=  | CrawlService, seed script          |
| =row_to_model()= | =utils/db.py=     | All repositories                   |
| =score_utils=   | =utils/scores.py=  | analyzer.py + batch process Lambda |

Import hierarchy: handlers → services → repositories → utils. No circular imports.

** No ORM

Raw psycopg2 only. No SQLAlchemy, no Peewee. The schema is simple and stable; an ORM adds
indirection without benefit given single-purpose Lambda functions and direct SQL control.

Games are analyzed in descending review count order. Three phases:

1. *Phase 1 — Top 500* (highest review count): pre-generate immediately after first crawl.
   Warms CloudFront cache. Covers ~80% of search traffic. ~$170 upfront.
2. *Phase 2 — Remaining 500+ review games* (~5,500 games): batch-process in background
   after Phase 1, or lazy on first request.
3. *Phase 3 — Long tail* (1–499 reviews, ~90k+ games): lazy on first request only.

All games get a report page. Low-review games get a "Limited data" badge and shorter analysis
(fewer chunks = less LLM cost). Cost scales with review count, not a flat fee (~$0.34/game max).

Lazy generation flow:
- First request for unanalyzed game → FastAPI triggers Step Functions → returns status page
- Client polls =/api/status/{job_id}= every 5s → delivers full report when analysis completes

* Ongoing Data Freshness

Four EventBridge scheduled rules maintain data quality after the initial seed:

| Rule             | Schedule          | Scope                          | Action                          |
|------------------+-------------------+--------------------------------+---------------------------------|
| nightly-top500   | Daily 6am UTC     | Top 500 by review_count        | Metadata + reviews + re-analyse |
| weekly-mid-tier  | Sundays 8am UTC   | review_count 500–5000          | Metadata + delta re-analyse     |
| monthly-long-tail| 1st of month      | review_count < 500             | Metadata only                   |
| weekly-discovery | Mondays 7am UTC   | Full Steam app list            | Discover and queue new games    |

*Delta-triggered re-analysis:* The app-crawler uses a tiered absolute threshold —
not a flat percentage — to decide whether to queue a review crawl + LLM analysis.
Large games (TF2: 800k reviews) need 10,000 new reviews to trigger; small games
(200 reviews) need just 25. This reflects the reality that big games have stable
sentiment while small games change meaningfully with every batch of new reviews.

*Staleness signal:* =last_analyzed= is included in all API responses and shown on the
frontend as "Analysis from X days ago". Reports older than 30 days show a "Refresh available"
badge.

*Cost estimate (steady state, post-seed with tiered delta thresholds):*
- Top 500 nightly: high-review games need 2k–10k new reviews to trigger → ~10 re-analyses/month × $0.25 ≈ ~$3
- Mid-tier weekly: ~200 games/month cross threshold × $0.10 ≈ ~$20
- Long tail monthly: metadata only, no LLM — negligible
- *Total ongoing LLM cost: ~$25–50/month*
- Compare: flat 5% threshold ≈ ~$500–700/month; naive nightly everything ≈ ~$2,700/month

** Manual Trigger Script (=trigger_crawl.py=)

EventBridge schedules are deployed =enabled=False= — enable only after initial seed
and site launch. Until then (and for ad-hoc operations), use =trigger_crawl.py= to
publish the same SNS events the automated pipeline uses.

Two subcommands:

*=crawl=* — publishes =GameMetadataReadyEvent= to the game-events SNS topic:
#+BEGIN_SRC sh
# Single game (dry-run):
poetry run python scripts/trigger_crawl.py crawl --appid 440 --dry-run

# All games with 500+ Steam reviews but 0 crawled reviews:
poetry run python scripts/trigger_crawl.py crawl --needs-reviews --limit 20

# Override the eligibility threshold:
poetry run python scripts/trigger_crawl.py crawl --needs-reviews --threshold 200 --limit 50
#+END_SRC

*=analyze=* — publishes =ReviewsReadyEvent= to the content-events SNS topic:
#+BEGIN_SRC sh
# Single game:
poetry run python scripts/trigger_crawl.py analyze --appid 440 --dry-run

# Games with reviews but no report:
poetry run python scripts/trigger_crawl.py analyze --needs-report --limit 10

# Games with stale reports (older than 30 days):
poetry run python scripts/trigger_crawl.py analyze --stale-days 30
#+END_SRC

Both subcommands support =--dry-run= (shows table of matched games, skips SNS publish)
and =--limit N= (default 50). Requires =DATABASE_URL= and the relevant topic param name env var
(=GAME_EVENTS_TOPIC_PARAM_NAME= or =CONTENT_EVENTS_TOPIC_PARAM_NAME=).

* Report Schema

** LLM synthesis output (Pass 2 — Sonnet)

Each section answers a distinct question with a distinct output format. Explicit exclusion rules
prevent duplication across sections.

#+BEGIN_SRC json
{
  "game_name": "string",
  "appid": 12345,
  "total_reviews_analyzed": 347,

  "overall_sentiment": "Overwhelmingly Positive|Very Positive|Positive|Mixed|Negative|Very Negative|Overwhelmingly Negative",
  "sentiment_score": 0.82,
  // Calculated in Python from batch stats — never LLM-guessed.
  // Thresholds: ≥0.95 Overwhelmingly Positive, ≥0.80 Very Positive, ≥0.65 Positive,
  //             ≥0.45 Mixed, ≥0.30 Negative, ≥0.15 Very Negative, <0.15 Overwhelmingly Negative

  "sentiment_trend": "declining",
  // Direction of change: 'improving' | 'stable' | 'declining'
  // DISTINCT FROM sentiment_score: a game can have 0.82 score that is declining.
  "sentiment_trend_note": "Recent reviews increasingly cite bot infestation; older reviews mostly positive.",
  // WHY the trend is happening — not a restatement of the trend value.

  "one_liner": "Timeless class-based shooter undermined by years of developer neglect and rampant bots.",
  // Max 25 words. Gamer-facing purchase decision. The ONLY section a gamer needs to read.

  "audience_profile": {
    "ideal_player": "Competitive FPS fans who love deep character mastery and community-driven play",
    "casual_friendliness": "low",
    "archetypes": ["hat collectors", "competitive tryharders", "longtime loyalists"],
    "not_for": ["players who hate grind", "newcomers expecting a polished onboarding experience"]
    // NOT_FOR is identity-based ('who shouldn't buy'), not moment-based (that's churn_triggers)
  },

  "design_strengths": [
    "Class system creates natural role differentiation that rewards team coordination"

** Section anti-duplication rules

The same root cause CAN appear in multiple sections ONLY if each instance describes a different aspect:
- =gameplay_friction=: describes the design flaw ("matchmaking has no skill separation")
- =churn_triggers=: describes WHEN it causes player loss ("new players encounter bots in minute one")
- =dev_priorities=: prescribes the fix ranked by ROI ("deploy bot detection")

If the same SENTENCE or FRAMING appears in two sections — that is duplication. Remove it from the section whose definition fits least.

** Freemium Philosophy

Two clean conceptual tiers:
- *Free = "What players think"* — consumer-facing intelligence, SEO value, useful to anyone
- *Pro = "What you should do about it"* — developer-facing action intelligence, worth paying for

Each free section is designed to naturally create desire for the adjacent pro section:
- Design Strengths → sets up "what should I build next?" → Player Wishlist
- Gameplay Friction → sets up "why are people leaving?" → Churn Triggers
- Audience Profile → sets up "how does this compare?" → Competitive Context
- Sentiment Trend → sets up "what changed and why?" → deeper via Pro

*No rate limiter on free content.* The free report is unconditional — no account, no IP limit,
no friction. The public site is a sales funnel, not a profit center.

** SEO requirements

- H1 = game name always
- H2s = "What Players Love", "Common Complaints", "Who Should Play This"
- Next.js SSR guarantees Googlebot sees full content on first request
- Structured data: =VideoGame= schema + =Review= aggregate
- Canonical URLs, 301 redirects on slug changes
- Sitemap with =<lastmod>= set to =last_analyzed= date
- OG tags: game cover image + one-liner

** Cross-linking strategy

Three directions from every page:
1. *Horizontal*: competitive_mentions → bidirectional game→game links
2. *Up*: breadcrumb to genre/tag index
3. *Down*: genre/tag/developer pages link back to all games in that group

** Content moat: index pages

Genre, tag, and developer pages aggregate insights across all games in that group:
- "The #1 complaint in Survival Horror is checkpoint frequency (37% of games)"
- Hundreds of additional high-value pages beyond the game report pages.

* Monetization

** Decision: No display ads

- Gaming audience has 40-60% ad-block rate
- AdSense CPMs for gaming: $1-3 (not meaningful until 500k+ monthly visits)
- Ads undermine premium conversion at exactly the wrong moment
- The public site is a *sales funnel*, not a profit center
