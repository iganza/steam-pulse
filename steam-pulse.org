** Executive Summary

** Project Description

Steam hosts over 100,000 games with millions of player reviews and game descriptions, but this
wealth of qualitative feedback and information remains largely untouched.  Developers, publishers and
investors rely on aggregate scores and manual reading to understand player sentiment -- a process
that doesn't scale and misses the deeper patterns buried across thousands of reviews.

The project proposes a batch LLM-powered pipeline that ingests the Steam game catalog, game
descriptions, reviews, and other relevant data, the processes them through Amazon Bedrock foundation
models to extract, structured, query-able insights.  A two-pass approach extracts summary
information first, which is then processed by collected and saved, and provided to a further LLM
processing step responsible for identifying relevant information such as recurring complaints,
praised features, and competitive comparison across the entire catalog.

The result is a structured intelligence layer on top of raw data, query-able via simple SQL or
optionally enhanced with semantic search.  This enabled questions that were previously impossible at
scale:  "What do players consistently criticize about survival games released in the last two
years?" or "How does this studio's latest title compare to competitors across various sentiment
dimensions".

The target market includes:  indie and AA game studios seeking actionable feedback, publishers
evaluating acquisitions or portfolio health, gaming-focused investors performing due diligence, and
market research firms looking to add qualitative sentiment data to their offerings.

No existing tool in the space provides this depth of LLM-extracted intelligence.  Current solutions
stop at surface-level metrics like scores, player counts, and revenue estimates.

*** Smart Game Selection — 500 Candidates
    Do NOT just pick by review count — TF2 has 800k reviews but is stale/irrelevant.
    Goal: games people are actively searching for and buying TODAY.

    Run this SQL after metadata crawl completes to get 500 well-ranked candidates:

    #+begin_src sql
    -- Score each game by recency + review volume + sentiment quality
    SELECT
        g.appid,
        g.name,
        g.release_date,
        g.review_count_english,
        g.positive_pct,
        g.review_score_desc,
        g.price_usd,
        g.is_free,
        (
            -- Recency boost: games released in last 3 years score highest
            CASE
                WHEN g.release_date >= NOW() - INTERVAL '1 year'  THEN 40
                WHEN g.release_date >= NOW() - INTERVAL '2 years' THEN 30
                WHEN g.release_date >= NOW() - INTERVAL '3 years' THEN 20
                WHEN g.release_date >= NOW() - INTERVAL '5 years' THEN 10
                ELSE 0
            END
            -- Review volume (log scale so TF2 doesn't dominate)
            + LEAST(CAST(LOG(GREATEST(g.review_count_english, 1)) * 5 AS INT), 30)
            -- Sentiment quality bonus (sweet spot: 70-95% positive = interesting)
            + CASE
                WHEN g.positive_pct BETWEEN 70 AND 95 THEN 20
                WHEN g.positive_pct BETWEEN 60 AND 70 THEN 10
                WHEN g.positive_pct > 95 THEN 10
                ELSE 0
              END
            -- Hidden gem bonus (high sentiment, under 5000 reviews)
            + CASE
                WHEN g.review_count_english < 5000 AND g.positive_pct >= 85 THEN 15
                ELSE 0
              END
        ) AS relevance_score
    FROM games g
    WHERE
        g.type = 'game'
        AND g.review_count_english >= 500
        AND g.release_date >= '2018-01-01'
        AND g.coming_soon = FALSE
        AND g.review_score_desc NOT IN ('Overwhelmingly Negative', 'Very Negative')
    ORDER BY relevance_score DESC, g.review_count_english DESC
    LIMIT 500;
    #+end_src

    After getting the 500, manually exclude:
    - Games with primarily non-English audiences
    - Abandonware / delisted games
    - TF2, CSGO, Dota2 (old F2P giants — everyone knows them, low SEO value)

** Multi-Region Crawl Architecture
   Goal: Distribute Steam API calls across N AWS regions to bypass per-IP
   rate limits. Each region provides a distinct IP pool. Only the primary
   region has the database, SNS topics, and application stack.

*** Design: Hub-and-Spoke with SQS Fan-Out

    #+begin_example
    ┌─── Primary Region (us-west-2) ──────────────────────────────┐
    │                                                              │
    │  ┌──────────┐    ┌──────────────┐    ┌──────────────────┐   │
    │  │ Dispatch │───▶│ SQS Queues   │    │ RDS (PostgreSQL) │   │
    │  │ Lambda   │    │ app_crawl_q  │    └────────▲─────────┘   │
    │  └──────────┘    │ review_crawl │             │              │
    │       │          └───────┬──────┘             │              │
    │       │                  │                    │              │
    │       │          ┌───────▼──────┐             │              │
    │       │          │ Crawl Lambda │─────────────┘              │
    │       │          │ (local work) │  writes to DB directly     │
    │       │          └──────────────┘                            │
    │       │                                                      │
    │       │  EventBridge  ┌───────────────────┐                  │
    │       └──────────────▶│ Cross-Region Bus  │                  │
    │                       └─────┬─────┬───────┘                  │
    └─────────────────────────────┼─────┼──────────────────────────┘
                                  │     │
              ┌───────────────────┘     └──────────────────┐
              ▼                                            ▼
    ┌─── Spoke: us-east-1 ──────────┐   ┌─── Spoke: eu-west-1 ──────┐
    │                                │   │                            │
    │  ┌────────────┐  ┌──────────┐ │   │  ┌────────────┐  ┌──────┐ │
    │  │ SQS Queue  │──│ Crawl    │ │   │  │ SQS Queue  │──│Crawl │ │
    │  │ (regional) │  │ Lambda   │ │   │  │ (regional) │  │Lambda│ │
    │  └────────────┘  └────┬─────┘ │   │  └────────────┘  └──┬───┘ │
    │                       │       │   │                      │     │
    │         ┌─────────────┘       │   │        ┌─────────────┘     │
    │         ▼                     │   │        ▼                   │
    │  EventBridge reply            │   │  EventBridge reply         │
    │  → primary region             │   │  → primary region          │
    │  (crawl results)              │   │  (crawl results)           │
    └───────────────────────────────┘   └────────────────────────────┘
    #+end_example

*** Spoke stacks contain ONLY:
    - 1 SQS queue (receives work items from primary via EventBridge)
    - 1 Crawl Lambda (calls Steam API, returns results via EventBridge)
    - 1 Lambda Layer (library_layer code — Steam client, models)
    - NO database, NO RDS, NO SNS topics, NO VPC peering

*** Data flow
    1. Primary "Dispatch Lambda" partitions appids across regions
       (round-robin or by appid hash mod N)
    2. Sends =crawl-work= events to EventBridge → routed to each region's SQS
    3. Each spoke Lambda fetches data from Steam API (distinct IP per region)
    4. Spoke Lambda sends =crawl-result= event back via EventBridge
       containing the full API response payload
    5. Primary "Ingest Lambda" receives results, writes to RDS, publishes SNS

*** Why EventBridge not cross-region SQS
    - EventBridge has native cross-region event routing (Event Bus → Event Bus)
    - SQS does not support cross-region natively (would need SNS fan-out)
    - EventBridge is $1/million events — crawling 160k games = $0.16

*** CDK implementation: SpokeStack
    One CDK stack, deployed to N regions. Minimal:

    #+begin_src python
    class CrawlSpokeStack(Stack):
        """Lightweight crawl worker — deploy to any region for more IP diversity."""

        def __init__(self, scope, id, primary_region: str, **kwargs):
            super().__init__(scope, id, **kwargs)

            # Regional queue — receives work from primary EventBridge
            queue = sqs.Queue(self, "CrawlQueue",
                visibility_timeout=Duration.minutes(10),
                retention_period=Duration.days(1))

            # Crawl Lambda — same code, no DB access
            fn = lambda_.Function(self, "CrawlWorker",
                runtime=lambda_.Runtime.PYTHON_3_12,
                handler="crawl_spoke.handler",
                code=lambda_.Code.from_asset("src/lambda-functions"),
                timeout=Duration.minutes(5),
                memory_size=256)
            fn.add_event_source(SqsEventSource(queue, batch_size=10))

            # EventBridge rule — send results back to primary region
            bus = events.EventBus.from_event_bus_arn(self, "PrimaryBus",
                f"arn:aws:events:{primary_region}:{self.account}:event-bus/steampulse")
            # Lambda publishes to this bus after each crawl
    #+end_src

*** "What genre should I concentrate on right now?"
    Multi-signal answer combining data we already collect:

    - *Supply gap:* Genres with growing review velocity but few new releases = underserved
    - *Sentiment quality:* Genres where even mid-tier games score "Very Positive" = forgiving audience
    - *Wishlist density:* =player_wishlist= items repeating across many reports = unmet demand
    - *Hidden gem ratio:* High % of hidden gems = good games getting buried = opportunity
    - *Review velocity trend:* Is the genre growing, peaking, or declining?

    Surfaced as an *Opportunity Score per genre* once we have 500+ analyzed games.

    Example output:
    #+begin_example
    Roguelike Deckbuilders: 🔥 High opportunity

    — Top wishlist: "endless run mode", "more card synergies", "co-op"
    — 3 hidden gems under 1k reviews
    — Review velocity: growing +18% YoY
    #+end_example

** V2 Chat Architecture — Bedrock Knowledge Bases
   The V2 =/chat= endpoint will use AWS Bedrock Knowledge Bases (RAG) to answer
   natural language questions over our entire dataset. This replaces building a
   custom NL→SQL pipeline.

*** Hybrid KB approach
    Two knowledge bases working together:

    - *Structured KB* → pointed at our RDS PostgreSQL.
      Bedrock auto-generates SQL for countable/filterable questions:
      "How many roguelikes released in 2025?", "Top 10 by sentiment", etc.

    - *Unstructured KB* → 6,000 game report JSONs + chunk summaries in S3.
      RAG retrieval for semantic/thematic questions:
      "What frustrates RPG players?", "Compare community health of TF2 vs CS2"

    Together these answer ~90 of the Top 100 questions with zero custom query code.

*** What it does NOT replace
    - Two-pass analyzer (generates the reports — KB consumes them)
    - PostgreSQL full-text search (faster/cheaper for exact keyword search)
    - Analytics endpoints (pre-computed stats faster than RAG for dashboards)

*** Infrastructure
    - OpenSearch Serverless: ~$350/month minimum
    - Embedding 6,000 reports: ~$5 (cheap)
    - Embedding 12M raw reviews: ~$500+ (unnecessary — reports summarize them)
    - Wired via =bedrock-agent-runtime:retrieve-and-generate= API

*** Prerequisites before implementation
    1. All 6,000 game reports generated and stored in DB
    2. Export reports to S3 as JSON documents
    3. Create Bedrock KB pointing at S3 bucket + RDS
    4. Wire =/chat= endpoint to Bedrock retrieve-and-generate
    5. The Top 100 questions become the test/eval suite

*** Data gaps to note (not blocking V2, just won't have answers)
    - No DLC↔base game relationship
    - No game engine data
    - No discount/price history
    - No concurrent player counts

* Analytics & Charts — Navigation Reference

Quick reference for where each chart lives in the UI.

** /analytics — Catalog-Wide Dashboard

Main nav → "Analytics" link (BarChart3 icon). Shows 9 time-series charts across
the entire Steam catalog. Control bar (granularity + genre filter) is Pro-gated.

- Release Volume — bar + moving average, sentiment overlay
- Sentiment Distribution — stacked area (positive / mixed / negative), Metacritic line
- Genre Share — stacked area of top genres over time
- Review Velocity — stacked bars by velocity band (<1/day, 1–10, 10–50, 50+)
- Pricing Trends — free % bars + avg paid price line
- Early Access Trends — EA count/% bars + EA vs non-EA sentiment lines
- Platform & Steam Deck — Mac %, Linux %, Deck Verified/Playable/Unsupported %
- Engagement Depth — stacked area by playtime bucket (<2h, 2–10h, 10–50h, 50–200h, 200h+)
- Feature Adoption — multi-line Steam feature adoption (Workshop, VR, Co-op, etc.)

** /games/[appid]/[slug] — Individual Game Report

Scroll below the report text. Charts are in the =GameAnalyticsSection= block.

- Playtime vs Sentiment — correlation chart
- Review Velocity — area chart, 24-month trend
- Early Access Impact — EA vs post-launch sentiment/playtime comparison
- Top Reviews — most helpful/funny review cards
- Audience Overlap — similar games by shared reviewers (5 free, all on Pro)

** /genre/[slug] — Genre Pages

e.g. =/genre/action=, =/genre/rpg=. Charts appear below the game list.

- Price Positioning — price distribution + sentiment by price range
- Release Timing — release density by month-of-year
- Platform Gaps — Windows / Mac / Linux support percentages

** /tag/[slug] — Tag Pages

e.g. =/tag/multiplayer=, =/tag/roguelike=. Chart appears below the game list.

- Tag Trend — yearly game count + avg sentiment for the tag

** /developer/[slug] — Developer Profile Pages

e.g. =/developer/valve=. Chart appears alongside the game list.

- Developer Portfolio — sentiment and review trends across all games by that developer
