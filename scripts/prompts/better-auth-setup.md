# Better Auth setup

Wire **Better Auth** as the identity provider for SteamPulse: magic link + Google OAuth + logout + account management, with the user database living in our own Postgres alongside `subscriptions` and `report_purchases`. This prompt is the implementation spec for the auth half of launch-plan Step 2 (the Stripe + Resend half is `scripts/prompts/stripe-resend-setup.md`).

The shape this delivers is defined in `scripts/prompts/rdb-launch-spec.md` Section 5. This prompt makes it concrete and actionable.

## Why Better Auth

Open-source TypeScript-first auth framework, current maintainer of Auth.js (NextAuth) and the recommended starting point for new Next.js projects in 2026. Plugin-based architecture covers magic link, Google OAuth, passkeys, 2FA, organisations, and account UI patterns while keeping the user database in our Postgres. Zero recurring cost, zero vendor lock-in, no migration ever required because the user table is ours.

Sources:
- Docs: https://better-auth.com/
- Next.js integration guide: https://better-auth.com/docs/integrations/next
- Stewardship announcement: https://better-auth.com/blog/authjs-joins-better-auth

## Files to create / modify

### Database migration

`src/lambda-functions/migrations/<next>__better_auth_tables.sql`:

```sql
-- depends: 0057_waitlist_confirmation_sent_at
-- transactional: true

-- Better Auth core tables. Schema mirrors Better Auth's required shape so the
-- adapter can read/write directly. See https://better-auth.com/docs/concepts/database
-- for the canonical reference. This migration is hand-written rather than
-- generated by the Better Auth CLI so it lives in version control next to
-- every other schema change.

CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email TEXT UNIQUE NOT NULL,
    email_verified BOOLEAN NOT NULL DEFAULT false,
    name TEXT,
    image TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token TEXT UNIQUE NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    ip_address TEXT,
    user_agent TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX sessions_user_idx ON sessions (user_id);
CREATE INDEX sessions_token_idx ON sessions (token);

CREATE TABLE accounts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    account_id TEXT NOT NULL,           -- provider's user id (e.g., Google sub)
    provider_id TEXT NOT NULL,          -- "google", "magic-link", etc.
    access_token TEXT,
    refresh_token TEXT,
    id_token TEXT,
    expires_at TIMESTAMPTZ,
    password TEXT,                      -- hashed; null for OAuth-only users
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (provider_id, account_id)
);
CREATE INDEX accounts_user_idx ON accounts (user_id);

CREATE TABLE verifications (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    identifier TEXT NOT NULL,           -- email for magic links
    value TEXT NOT NULL,                -- hashed token
    expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX verifications_identifier_idx ON verifications (identifier);
```

After Better Auth ships and the implementation is stable, run `npx @better-auth/cli generate` against the live config to verify the hand-written schema matches what the framework expects. Adjust if the CLI flags differences (e.g., new optional columns added by future Better Auth versions); v1 ships with the schema above.

### Server-side auth helper for genre page render-mode decision

In `frontend/app/genre/[slug]/page.tsx`:

```ts
import { auth } from "@/lib/auth";
import { headers } from "next/headers";

export default async function GenrePage({ params }: Props) {
  const session = await auth.api.getSession({ headers: await headers() });
  const userId = session?.user?.id ?? null;

  const isPaid = userId
    ? await checkEntitlement(userId, slug)  // queries subscriptions + report_purchases
    : false;
  // render free or paid mode based on isPaid
}
```

`checkEntitlement(userId, reportSlug)` lives in `frontend/lib/entitlements.ts` (new). It runs the two queries from spec Section 5 against the same Postgres connection.

### Buy block flow

In `frontend/components/genre/ReportBuyBlock.tsx`: when the user clicks Subscribe or Buy, check `useSession()` from the auth client. If signed in, post to `/api/checkout/start` (which uses `auth.api.getSession({ headers })` server-side and creates a Stripe Checkout Session with `metadata.user_id`). If not signed in, redirect to `/sign-in?redirect=/genre/${slug}` first; on successful sign-in, the redirect query param brings them back and they re-click.

The webhook handler reads `metadata.user_id` from Stripe events and inserts the entitlement row keyed on `users.id`. No Better Auth API call required during webhook processing. Schema and webhook details: `scripts/prompts/stripe-resend-setup.md`.

## Failure modes to handle explicitly

- **User changes email.** Better Auth's email-update flow re-verifies. The `users.id` (UUID) does not change, so entitlements survive.
- **OAuth account already linked to a different email.** Better Auth's default behaviour is to refuse the link; the user has to sign in with their original method first, then add the OAuth account in `/account`. Document this in the sign-in page copy if it surprises buyers.
- **Magic link expired or reused.** Better Auth's `verifications` table tracks single-use; expired or used tokens fail with a clear error. Sign-in page handles the error and prompts a fresh request.
- **Session theft.** Better Auth supports immediate session revocation (logout invalidates the row in `sessions`), unlike pure stateless JWT. v1 inherits this for free.
